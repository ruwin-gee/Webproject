Feedback page

Technical Description

HTML Structure

The page is built using the shared HTML template. The main content is divided into two columns using CSS Grid: a scrollable programme directory on the left and a feedback form on the right. Key HTML elements used include:

  • <fieldset> and <legend> to group Personal Details and Feedback Details
  • <select> for the programme dropdown and multi-select involvement list
  • <datalist> for the searchable topic suggestions input
  • <input type="date"> and <input type="datetime-local"> for visit date fields
  • <textarea> with maxlength="500" for the comments field

CSS Techniques

Page-specific styles are embedded in a <style> block within the HTML file, as permitted by the coursework specification. Techniques used include:

  • CSS Grid for the two-column layout on large screens
  • Custom scrollbar styling using ::-webkit-scrollbar for the programme scroll area
  • CSS transitions on form inputs for focus states (green glow effect)
  • Error class toggling for red border and error message visibility

JavaScript

JavaScript handles three interactive features:

  • Live character counter: An input event listener updates the counter below the textarea on every keystroke, showing remaining characters with colour warnings at 50 and 0 remaining.
  • Form validation: A validateField() function checks each required field on submit, adds/removes .error classes, and shows/hides inline error messages using .visible class toggling.
  • Submission confirmation: On successful validation, the form is hidden and a confirmation message is revealed using DOM manipulation.

Accessibility Considerations – Feedback Page

Accessibility was considered throughout the development of this webpage to ensure it is usable by a wide range of users, including those with disabilities and users relying on assistive technologies such as screen readers and keyboard navigation.

Semantic HTML Structure

Semantic elements such as <header>, <nav>, <main>, and <footer> were used to create a clear and meaningful structure.

This helps screen readers understand the layout and improves navigation for users.

Form Accessibility

All form inputs are properly linked with labels using the for attribute. Required fields are indicated using aria-required="true". This ensures users and assistive technologies can clearly identify input fields and required data.

Error Handling and Validation

Form validation provides real-time feedback, highlighting errors visually and displaying messages using role="alert". This ensures screen readers immediately notify users when an error occurs.

ARIA Attributes and Live Regions

ARIA attributes such as aria-label and aria-live="polite" are used for dynamic elements like the character counter and confirmation messages. This allows screen readers to announce updates without interrupting the user.

Keyboard Accessibility

All interactive elements such as buttons, links, and form fields are accessible using the keyboard. Users can navigate using the Tab and Enter keys, ensuring accessibility for users who cannot use a mouse.

Images and Alternative Text

All images include descriptive alt text. This allows screen readers to describe images to visually impaired users, ensuring content is understandable without visual access.

Colour and Contrast

High-contrast colours were used to improve readability. Error messages are communicated using both colour and text, ensuring accessibility for users with colour vision deficiencies.

Responsive Design

The page is fully responsive using CSS media queries, allowing it to function correctly on different screen sizes and devices, including mobile phones and tablets.

Conclusion

These accessibility features improve usability, inclusivity, and ensure that the website can be effectively used by a diverse range of users.

Team Page

Technical Description

HTML Structure

The team page uses a CSS Grid layout to display four member cards in a responsive grid — one column on mobile, two on tablet, and four on desktop. Each card uses a <div> with tabindex="0" and role="button" to ensure keyboard accessibility.

  • Circular photo wrapped in a <div> for the border and scaling effect
  • Expandable details panel using a hidden <div> with max-height: 0 and overflow hidden
  • All images include meaningful alt text describing the team member and their role

CSS Techniques

The Team Page uses only CSS for all interactivity, as permitted by the specification. Key techniques include:

  • Hover scaling: transform: scale(1.1) on the photo on :hover and :focus
  • Expandable panel: max-height transitions from 0 to 220px on hover, combined with opacity transition for a smooth reveal
  • Focus styling: outline: 3px solid #4ade80 on :focus for visible keyboard navigation state
  • Smooth transitions: All hover effects use transition with 0.3s ease timing

Content Page — E-Waste & Climate Pollution

Technical Description

HTML Structure

The content page covers the topic of e-waste pollution and its links to climate change. The page is structured with a sticky sidebar navigation and a main article area divided into six clearly labelled sections:

  • One <h1> in the hero banner for the page title
  • <h2> for each major section heading
  • <h3> for sub-headings within sections
  • Each section has a unique id attribute targeted by the sidebar anchor links
  • <aside> used for the sticky sidebar navigation
  • <article> used to wrap the main content body

CSS Techniques

  • position: sticky; top: 80px for the sidebar navigation
  • scroll-margin-top: 80px on each section to offset the fixed navbar when jumping via anchor links
  • position: fixed for the "Go to Top" button — no JavaScript required
  • CSS Grid for the stat card grid and image grid layouts
  • CSS custom counter (counter-reset / counter-increment) for the numbered action list

Internal Navigation

All sidebar links use anchor links (href="#section-id") to jump to sections within the same page. scroll-behavior: smooth is applied via CSS on the html element for smooth scrolling.

Accessibility Considerations – Content Page (E-Waste)

Accessibility was considered throughout the design of the content page to ensure that information about e-waste and climate change is accessible to all users, including those using assistive technologies such as screen readers and keyboard navigation.

Semantic Structure

The page uses semantic HTML elements such as <header>, <main>, <section>, <article>, and <aside> to organise content logically. This helps screen readers understand the page structure and improves navigation.

Internal Navigation (Anchor Links)

A sidebar navigation menu with anchor links (e.g. #what-is-ewaste) allows users to quickly jump to different sections of the page. Smooth scrolling improves user experience and orientation.

Keyboard Accessibility

All navigation links, buttons, and interactive elements such as the “Go to Top” button are accessible using the keyboard. Users can navigate using the Tab key and activate links using the Enter key.

Images and Alternative Text

All images include descriptive alt text explaining the visual content. This ensures that screen readers can convey the meaning of images to visually impaired users.

ARIA Labels

ARIA attributes such as aria-label are used on interactive elements like the “Go to Top” button to provide clear descriptions for screen reader users.

Readable Content and Typography

Text is structured using headings, subheadings, and paragraphs to improve readability. Font sizes, spacing, and line height are carefully adjusted to make long-form content easier to read.

Colour and Contrast

High-contrast colours are used between text and background to improve readability. Important sections such as highlights and statistics use contrasting colours to draw attention without reducing accessibility.

Responsive Design

The layout uses responsive design techniques such as CSS Grid and media queries to ensure the page adapts to different screen sizes, including mobile and tablet devices.

Visual Aids and Content Structure

Information is supported with visual elements such as images, statistic cards, and highlighted boxes. These improve understanding and help users scan content efficiently.

Conclusion

These accessibility features ensure that the content page is inclusive, easy to navigate, and usable for a wide range of users, supporting both readability and interaction.

Challenges & Lessons Learned

Challenge 1 — CSS max-height Transition on Team Page

Animating the expandable details panel on the Team Page was tricky because CSS transitions do not work on height: auto. I resolved this by using max-height with a fixed value (220px) as the expanded state, combined with an opacity transition. This creates a smooth reveal effect without needing JavaScript.

Challenge 2 — Form Validation Without Page Reload

Implementing client-side form validation that shows inline error messages without reloading the page required careful DOM manipulation. I created a reusable validateField() function that adds/removes CSS classes to control error visibility, which kept the code clean and easy to maintain.

Challenge 3 — Sticky Sidebar with Fixed Navbar

The sticky sidebar on the content page initially appeared beneath the fixed navbar when scrolling. I resolved this by setting top: 80px on the sticky element (matching the navbar height) and adding scroll-margin-top: 80px to each section so anchor links land in the correct position.

7. JANET Compliance

Appropriate and Ethical Content

All content across my pages is original, educational, and directly relevant to SDG 13: Climate Action. No content has been copied from external sources. The e-waste content page presents factual, research-based information that aligns with the educational purpose of the Janet network.

Responsible Use of External Resources

All images used are royalty-free and properly referenced in the References section of this Page Editor. No third-party content has been reproduced without attribution. External fonts (Google Fonts) are loaded from an approved CDN.

Data Protection (GDPR)

The Feedback form does not send data to any server — it is a client-side demonstration only. No personal data is collected, stored, or transmitted, ensuring compliance with GDPR and the Janet AUP requirement to avoid collecting personal data unnecessarily.

Security and Best Practices

The website uses no malicious code, no tracking scripts, and no unauthorised third-party libraries. All code is original or clearly attributed. The site follows secure web development principles as required by the Janet Acceptable Use Policy.

Accessibility Compliance

All pages follow WCAG 2.1 AA guidelines as required by the Public Sector Bodies Accessibility Regulations 2018, which applies to institutions using the Janet network. Semantic HTML, alt text, and keyboard navigation have been implemented throughout.
